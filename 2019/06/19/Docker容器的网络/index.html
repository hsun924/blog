Docker容器的网络

发表于 2019-06-19 | 分类于 Docker

容器的网络

网络模式

安装Docker时，它会自动创建三个网络，采用docker network ls可以查看

运行容器的时候可以采用–net来指定

none

顾名思义，就是没有网络，该模式关闭了容器的网络功能；这种网络模式下的容器适合一些不需要网络的应用，比如只需要写磁盘卷的批处理任务

执行ifconfig只能查看到lo

host

相当于VM中的桥接模式，与宿主机在同一个网络中，但没有独立IP地址。
采用此模式创建的容器将不会虚拟出自己的网卡、配置自己的IP等，而是使用宿主机的IP和端口。

众所周知，Docker使用了Linux的Namespaces技术来进行资源隔离，如PID Namespace隔离进程，Mount Namespace隔离文件系统，Network Namespace隔离网络等。

一个Network Namespace提供了一份独立的网络环境，包括网卡、路由、iptable规则等，都与其他的Network Namespace隔离。

一个Docker容器一般会分配一个独立的Network Namespace。但如果启动容器的时候使用host模式，那么这个容器将不会获得一个独立的Network Namespace，而是和宿主机共用一个Network Namespace。

当我们在容器中执行任何类似ifconfig命令查看网络环境时，看到的都是宿主机上的信息。而外界访问容器中的应用，则直接使用宿主机IP:端口即可，不用任何NAT转换，就如直接跑在宿主机中一样。但是，容器的其他方面，如文件系统、进程列表等还是和宿主机隔离的。

bridge

相当于VM的NAT模式，是Docker容器的默认网络模式，也是最常用的一种;容器使用独立network Namespace，并连接到docker0虚拟网卡（默认模式）。通过docker0网桥以及Iptables nat表配置与宿主机通信；此模式会为每一个容器分配Network Namespace、设置IP等，并将一个主机上的Docker容器连接到一个虚拟网桥上。

bridge模式详解

查看宿主机的网络环境，会发现有一个名为docker0的虚拟网桥，它是Docker server启动时，在主机上创建；此主机上启动的Docker容器会连接到这个虚拟网桥上。虚拟网桥的工作方式和物理交换机类似，这样主机上的所有容器就通过交换机连在了一个二层网络中。接下来就要为容器分配IP了，Docker会从RFC1918所定义的私有IP网段中，选择一个和宿主机不同的IP地址和子网分配给docker0，连接到docker0的容器就从这个子网中选择一个未占用的IP使用。
如一般Docker会使用172.17.0.0/16这个网段，并将172.17.0.1/16分配给docker0网桥（在主机上使用ifconfig命令是可以看到docker0的，可以认为它是网桥的管理接口，在宿主机上作为一块虚拟网卡使用）。单机环境下的网络拓扑如下

Docker完成以上网络配置的过程大致是这样的：

  1. 在主机上创建一对veth pair接口。veth接口总是成对出现的，它们组成了一个数据的通道，数据从一个接口进入，就会从另一个接口出来。

  2. Docker将veth pair接口的一端放在新创建的容器中，并命名为eth0。另一端在主机中并被挂载到docker0网桥，并以veth开头(veth917df89)这样的名字命名;可以通过brctl show命令查看。

  3. 从docker0子网中分配一个IP给容器使用，并设置docker0的IP地址为容器的默认网关。

通过这种方式，主机可以跟容器通信，容器之间也可以相互通信，Docker就创建一个了在主机和所有容器之间虚拟共享网络

举一个例子说明一下，假设主机有一块网卡为eth0，IP地址为10.211.55.11/24，网关为10.211.55.1。从一个IP为172.17.0.1/16的容器中ping百度。IP包首先从容器发往自己的默认网关docker0，包到达docker0后，也就到达了主机上。然后会查询主机的路由表，发现包应该从主机的eth0发往主机的网关10.10.105.1/24。接着包会转发给eth0，并从eth0发出去。这时候，上面的iptables规则就会起作用，对包做SNAT转换，将源地址换为eth0的地址。这样，在外界看来，这个包就是从10.211.55.11上发出来的，Docker容器对外是不可见的。

自定义网桥

启动 Docker 时，会自动创建默认的桥接网络，也就是我们所说的docker0,新启动的容器如果没有特别指定都会连接到这个默认桥接网络。在同一网桥下的docker容器是默认互通的，除非特别指定（可以禁止它们之间通信，方法是在DOCKER_OPTS变量中设置–icc=false，这样只有使用–link才能使两个容器通信）；

如果说出于安全考虑，一般情况下建议使用自定义的网桥来控制哪些容器可以相互通信。

用户可以通过docker network create来创建和配置自定义网桥，驱动程序使用 -d [bridge/macvlan] 来指定，本次我们使用bridge

增加子网选项–subnet=172.20.0.0/16
